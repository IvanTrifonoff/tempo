# 🏗 Архитектура системы (Tempo v1.0.46)

## 1. Схема взаимодействия
Приложение построено по классической модели SPA (Single Page Application) с REST API.

- **Frontend:** React (TypeScript). Стейт-менеджмент через `useState` и `useCallback`. Audio API используется для метронома и воспроизведения.
- **Backend:** Express.js. Модульная структура:
    - `server.js` — точка входа и middleware.
    - `routes/` — описание эндпоинтов.
    - `controllers/` — бизнес-логика.
    - `db/` — подключение к PostgreSQL и пул соединений.
- **Storage:** Файлы треков хранятся локально в `server/uploads/`. В базе хранятся только пути.

## 2. Ключевые механизмы

### 🎵 Аудио и Метроном
После удаления Clap Detection (v1.0.46), Audio API используется в упрощенном режиме:
- `AudioContext` инициализируется при первом взаимодействии пользователя (Play или включение метронома).
- Метроном работает через синтез коротких импульсов (Oscillator) в реальном времени, синхронизированных с BPM трека.

### 🛡 Безопасность и Загрузка
- **Multer Filter:** В `middleware/upload.js` настроен белый список расширений и лимит 20MB.
- **UUID:** Все новые ID (пользователи, треки, плейлисты) — это UUID v4, что исключает перебор ID.
- **JWT:** Авторизация через JSON Web Token. Срок жизни не ограничен (для удобства тренеров), но может быть изменен в `.env`.

### 🔄 Синхронизация версий
- При запуске контейнера выполняется `node scripts/syncChangelog.js`.
- Скрипт парсит `CHANGELOG.md`, сопоставляет версию с `constants.tsx` и обновляет таблицу `changelogs`. 
- Это позволяет фронтенду показывать уведомление "Что нового?" сразу после деплоя.

## 📦 Docker Структура
- `Dockerfile` (Multi-stage):
    1. **Builder:** Сборка фронтенда (`npm run build`).
    2. **Runner:** Node.js сервер, раздающий статику из `/dist` и обрабатывающий API запросы.
